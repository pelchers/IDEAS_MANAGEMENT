# Audit Report — Database
**Run**: 001.04 | **Date**: 2026-04-08 | **Score**: 72 / 100 (C)

## Summary
The Prisma schema is well-structured with consistent cascade deletes, appropriate indexes on foreign key columns, and transaction usage for multi-step auth operations. The main concerns are: (1) `SyncOperation.status` is a plain `String` rather than a proper Prisma enum, risking invalid values; (2) the `ai/sessions` and `auth/me/export` routes have unbounded `findMany` queries with no `take` limit; (3) connection pool size is not configured for serverless deployment; (4) several models lack `updatedAt` timestamps. Overall the schema is solid for an early-stage project, but the missing enum and unbounded queries need attention before scale.

### HIGH

| ID | Severity | File | Finding |
|---|---|---|---|
| D-01 | High | `prisma/schema.prisma:347` | `SyncOperation.status` is typed as `String @default("pending")` with a comment listing valid values `// pending, applied, conflict, resolved`. This should be a Prisma `enum SyncStatus` to enforce validity at the DB level. Any typo or unknown value can silently persist with no validation. |
| D-02 | High | `src/app/api/auth/me/export/route.ts:33`, `src/app/api/ai/sessions/route.ts:14` | Two unbounded `findMany` queries: `aiChatSession.findMany({ where: { userId } })` returns all sessions for a user with no `take` limit. A user with thousands of sessions will cause this query to return all rows, potentially OOM-ing the serverless function. Same issue in the data export endpoint which also fetches all messages for all sessions. |

In serverless/edge deployments (Vercel), default Prisma pool sizes can exhaust PostgreSQL connection limits. | D-04 | Medium | `prisma/schema.prisma` | Several models lack `updatedAt`: `Session` (lines 79-91), `RefreshToken` (92-107), `AiChatMessage` (242-255), `AiToolOutput` (257-272), `ProjectMember` (308-321), `SyncSnapshot` (359-374), `AiTokenUsage` (376-391). | D-05 | Medium | `prisma/schema.prisma` | The `AiTokenUsage` model uses a `@@unique([userId, periodStart])` constraint but `periodStart` is a raw `DateTime` — period boundaries computed in application code could drift by milliseconds across different server instances, creating duplicate records instead of upserts, which would cause the `create-or-update` logic in `token-tracking.ts` to behave unexpectedly. |

## Recommendations
1. Convert `SyncOperation.status` to a Prisma `enum SyncOperationStatus` with values `PENDING`, `APPLIED`, `CONFLICT`, `RESOLVED` and create a migration.
2. Add `take` limits (e.g., `take: 100`) to the unbounded `findMany` queries in `/api/ai/sessions` and `/api/auth/me/export`, and add cursor-based pagination for the export endpoint.
3. Synchronise `ProjectStatusSchema` in `packages/schemas` to use uppercase values matching the Prisma enum, or add a transformation layer.
