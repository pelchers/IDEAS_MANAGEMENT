# Audit Report — API Contract
**Run**: 001.07 | **Date**: 2026-04-08 | **Score**: 58 / 100 (D)

## Summary
All protected API routes correctly use `requireAuth` or `requireAdmin` middleware. Error responses use a consistent `{ ok, error }` format, HTTP status codes are semantically correct (201 for creation, 401/403/404/429 used appropriately). The critical gaps are: no API versioning strategy, no rate limit headers on any response, most list endpoints are unbounded (no pagination), the SDK package is empty (`export {}`), and 70% of routes lack Zod request body validation. The `/api/auth/signup` endpoint also exposes a verification token in every response, which is an API contract issue as well as a security concern.

### HIGH

| ID | Severity | File | Finding |
|---|---|---|---|
| API-01 | High | All API routes | No API versioning strategy — no `/v1/` prefix, no `API-Version` header, no version documentation. Breaking changes to existing endpoints will silently affect clients. The SDK package (`packages/sdk/src/index.ts`) exports nothing (`export {}`), providing no typed client interface. |
| API-02 | High | `/api/ai/sessions/route.ts`, `/api/projects/route.ts`, `/api/auth/me/export/route.ts`, `/api/ai/tools/route.ts` | Multiple list endpoints have no pagination (`limit`, `offset`, `cursor`). As data grows, these queries will return increasingly large payloads with no truncation. The sessions list and full export endpoints are the highest risk. |
| API-03 | High | All API routes | No rate limit headers (`X-RateLimit-Limit`, `X-RateLimit-Remaining`, `Retry-After`) on any response. No rate limiting middleware at all. Auth endpoints are particularly exposed. |

### MEDIUM

| ID | Severity | File | Finding |
|---|---|---|---|
| API-04 | Medium | 26 of 37 routes | Missing Zod request validation on input bodies. Routes like `/api/ai/chat`, `/api/projects`, `/api/projects/[id]`, `/api/ai/config` rely on manual type checks (`typeof body.name !== "string"`) rather than a schema. This is not documented as dev-only in any API docs, creates a confusing response shape in production, and is a security issue. The contract should differ between environments. |
| API-06 | Medium | `/api/projects/route.ts:173` | The GET projects endpoint accepts `search`, `sort`, `order`, `status`, `tag` query parameters with no validation or sanitisation — an unknown `sort` value silently falls back to `updatedAt`, and an unknown `status` is silently ignored. Callers receive no feedback when they pass invalid parameters. |

## Recommendations
1. Establish an API versioning strategy — at minimum, document the current API as v1 and add a plan for breaking change handling.
2. Add pagination (`take`, `skip`, `cursor`) to all list endpoints. Start with `/api/ai/sessions` and `/api/projects`.
3. Create a shared `validateBody(schema, req)` utility that all routes use for Zod validation, and apply it to the 26 currently unvalidated routes.
