### MEDIUM

| ID | Severity | File | Finding |
|---|---|---|---|
| DT-02 | Medium | `packages/schemas/src/index.ts:3-7` vs `prisma/schema.prisma:159-164` | Enum value mismatch: `ProjectStatusSchema` in the shared packages uses lowercase (`"planning"`, `"active"`, `"paused"`, `"archived"`) while the Prisma `ProjectStatus` enum uses SCREAMING_CASE (`PLANNING`, `ACTIVE`, `PAUSED`, `ARCHIVED`). Any code that reads a DB project status and validates it with `ProjectStatusSchema` will fail. |
| DT-03 | Medium | `src/server/ai/token-tracking.ts:71-72` | Token usage periods computed as `new Date(now.getFullYear(), now.getMonth(), 1)` using local time, but server time zone in production (Vercel, typically UTC) may differ from the period computed at record creation time. If two requests arrive at the UTC month boundary from different time zones, two records could be created for the same `userId` ‚Äî which the `@@unique([userId, periodStart])` constraint would correctly reject, but the error would be uncaught, silently dropping usage tracking. |
| DT-04 | Medium | `prisma/schema.prisma` (absent) | No seed data file (`prisma/seed.ts` or `seed.sql`). There is no defined set of test/development records. Developers must manually create admin users via the bootstrap script and projects via the UI. No repeatable baseline state for development or testing. |

## Recommendations
1. Convert `SyncOperation.status` to a Prisma enum (`SyncOperationStatus`) with values `PENDING`, `APPLIED`, `CONFLICT`, `RESOLVED`. Create a migration.
2. Fix the enum case mismatch: update `packages/schemas/src/index.ts` `ProjectStatusSchema` to use uppercase values matching the Prisma enum.
3. Add a `prisma/seed.ts` file that creates test users, projects, and sample artifacts for development/test environments.
