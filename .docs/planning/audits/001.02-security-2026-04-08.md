# Audit Report — Security
**Run**: 001.02 | **Date**: 2026-04-08 | **Score**: 48 / 100 (D)

## Summary
Auth is implemented well (HttpOnly cookies, argon2id password hashing, refresh token rotation, webhook signature verification). However, there are three high-severity issues: security headers are entirely absent (no CSP, HSTS, X-Frame-Options), an open redirect vulnerability exists in the sign-in flow, and the signup endpoint unconditionally returns a raw email verification token in every response — including production responses. Rate limiting is completely absent across all API routes, representing meaningful DoS and credential-stuffing risk.

## Findings

### CRITICAL

| ID | Severity | File | Finding |
|---|---|---|---|
| S-01 | Critical | `src/app/api/auth/signup/route.ts:55` | Email verification token returned unconditionally in every signup response as `_dev.verificationToken`. There is no `NODE_ENV` guard. In a production deployment this exposes a privileged token in every JSON signup response, allowing any attacker who reads the response to verify any email they sign up with or replay the token. |
| S-02 | Critical | `next.config.ts` | No security headers configured: CSP, HSTS (`Strict-Transport-Security`), `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, and `Permissions-Policy` are all absent. Next.js does not add these by default. Absence of CSP enables XSS escalation; absence of X-Frame-Options enables clickjacking. |

### HIGH

| ID | Severity | File | Finding |
|---|---|---|---|
| S-03 | High | `src/app/signin/page.tsx:28-29` | Open redirect: the `redirect` query parameter is used directly as `window.location.href` without validation. An attacker can craft a link like `/signin?redirect=https://evil.com` and redirect authenticated users after login. |
| S-04 | High | All API routes | No rate limiting on any endpoint. Auth endpoints (signin, signup, password-reset, refresh) are fully unthrottled — enabling credential stuffing, brute force, and account enumeration attacks. No `X-RateLimit-*` headers present. |

### MEDIUM

| ID | Severity | File | Finding |
|---|---|---|---|
| S-05 | Medium | `src/server/auth/config.ts` | `AUTH_COOKIE_SECURE` defaults to `false` when the env var is unset or not exactly `"true"`. This means cookies are sent over plain HTTP in any environment where the variable is not explicitly set, including misconfigured staging environments. |
| S-06 | Medium | `src/app/(authenticated)/projects/[id]/whiteboard/page.tsx:158-185` | `dangerouslySetInnerHTML` used to render markdown content from user-uploaded `.md` files. The `renderMarkdownToHtml` function escapes HTML entities at the start, but then generates raw `<a href="$2">` link tags where `$2` is the URL from user content — `javascript:` protocol URIs are not filtered. |
| S-07 | Medium | `src/app/api/admin/grant-entitlement/route.ts:20-21` | The admin entitlement route performs no Zod validation on `body.feature` and `body.targetUserId`. The feature string is passed directly to Prisma upsert; while Prisma prevents injection, an arbitrary string creates an entitlement for a non-existent feature, which could confuse billing/access logic. |

### LOW

| ID | Severity | File | Finding |
|---|---|---|---|
| S-08 | Low | `src/app/api/auth/signup/route.ts:44` | Comment says "stub: would send email in production" indicating the email verification flow is not production-ready — users can use the app without ever verifying their email. |
| S-09 | Low | `src/server/db.ts` | No connection pool configuration explicitly set on PrismaClient. Default Prisma pool size is `num_physical_cpus * 2 + 1` which may be too high for serverless/Vercel deployments, exhausting database connections. |

## Recommendations
1. Add a `NODE_ENV === "production"` guard around the `_dev.verificationToken` in the signup response, or remove it and implement actual email delivery.
2. Add security headers in `next.config.ts` via the `headers()` async function, including CSP, HSTS, X-Frame-Options, X-Content-Type-Options.
3. Validate and restrict the `redirect` parameter to same-origin paths only before using it in `window.location.href`.
