I need to plan the schema migration for v2.15. We need to add a comments table and an avatar_url column to users. What's the best approach?

AI ASSISTANT 09:00

Here's my recommended approach for the v2.15 migration:

Step 1: Create the comments table

CREATE TABLE comments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  idea_id UUID NOT NULL REFERENCES ideas(id),
  author_id UUID NOT NULL REFERENCES users(id),
  body TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_comments_idea_id ON comments(idea_id);
CREATE INDEX idx_comments_author_id ON comments(author_id);

Step 2: Alter users table

ALTER TABLE users
  ADD COLUMN avatar_url VARCHAR(500) DEFAULT NULL;

Key considerations:

  • Run the ALTER TABLE in a separate migration from the CREATE TABLE to isolate rollback points
  • The avatar_url column is nullable, so no backfill needed
  • Add a foreign key cascade: ON DELETE CASCADE for comments when an idea is deleted
J. DOE 09:05

Good. Should we add soft delete to comments? And what about threading - should comments support replies?

AI ASSISTANT 09:05

Great questions. Here's my recommendation:

Soft delete: Yes, add a deleted_at TIMESTAMP DEFAULT NULL column. This allows recovery and maintains referential integrity. Filter with WHERE deleted_at IS NULL in queries.

Threading: Add a self-referencing parent_id UUID REFERENCES comments(id) column. This gives you unlimited nesting depth. For the UI, I'd recommend limiting display to 2 levels deep and showing a "view thread" link for deeper chains.

CREATE TABLE comments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  idea_id UUID NOT NULL REFERENCES ideas(id),
  author_id UUID NOT NULL REFERENCES users(id),
  parent_id UUID REFERENCES comments(id),
  body TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW(),
  deleted_at TIMESTAMP DEFAULT NULL
);

This keeps the schema flexible without over-engineering.
