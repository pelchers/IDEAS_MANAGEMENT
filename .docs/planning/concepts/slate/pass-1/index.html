import React from 'react';
import { useProjectMetrics } from '../hooks/useMetrics';
import { StatCard, BarChart } from './shared';

interface DashboardProps {
  projectId: string;
  timeRange?: '7d' | '30d' | '90d';
}

export const Dashboard: React.FC<DashboardProps> = ({
  projectId,
  timeRange = '7d'
}) => {
  const { metrics, loading } = useProjectMetrics(projectId, timeRange);

  if (loading) return <LoadingSkeleton />;

  return (
    <div className="dashboard-container">
      <StatGrid metrics={metrics.summary} />
      <BarChart data={metrics.throughput} />
      <ActivityFeed items={metrics.activity} />
    </div>
  );
};

We need to design a multi-tenant isolation strategy for Orion v2. The main concern is data leakage between tenants. What approaches should we consider?
14:02
JD
AI
For multi-tenant isolation in Orion v2, there are three primary strategies to evaluate: 1. Row-Level Security (RLS) Shared database, shared schema. Each row has a tenant_id column. PostgreSQL RLS policies enforce isolation at the database level. Lowest infrastructure cost, but requires careful policy management. 2. Schema-per-Tenant Shared database, separate schemas. Each tenant gets their own schema namespace. Better isolation than RLS, moderate complexity for migrations. 3. Database-per-Tenant Complete physical isolation. Highest security guarantee but significantly higher infrastructure and operational costs. Given the scale indicators you described (18 team members, ~24 projects), I'd recommend starting with RLS and designing the abstraction layer so you can migrate to schema-per-tenant later if compliance requirements demand it. Shall I detail the RLS implementation pattern?
14:02
Yes, outline the RLS approach. Include the middleware layer that sets the tenant context.
14:05
JD
AI
Here's the RLS implementation pattern: -- Enable RLS on all tenant-scoped tables ALTER TABLE projects ENABLE ROW LEVEL SECURITY; -- Create policy: users see only their tenant's data CREATE POLICY tenant_isolation ON projects USING (tenant_id = current_setting('app.tenant_id')::uuid); -- Middleware sets tenant context per request SET LOCAL app.tenant_id = 'tenant-uuid-here'; The middleware layer intercepts every request, extracts the tenant ID from the JWT claims, and sets the PostgreSQL session variable before any queries execute. This creates an airtight isolation boundary at the database engine level. Key considerations: - Always use SET LOCAL (transaction-scoped, not session-scoped) - Add tenant_id to all composite indexes - Implement integration tests that verify cross-tenant queries return empty results
