# branched-logs — NOTES

## Why
Multi-device append-only logs (chat history, HANDOFF) corrupt when both machines append to ONE file and
git-merges. Branched logs make writes physically disjoint (per-device segments) so git never conflicts,
and regenerate the merged view deterministically. **0 chat-history loss is the hard requirement.**

## Design decisions
- **Segments = branches.** `<base>.<device>.md` per device; `<base>.md` is a derived, regenerated view.
- **Hidden markers** `<!-- ENTRY ts= device= id= -->` — invisible when rendered, give deterministic sort
  (ts) + content-hash dedup (id). Shared-ancestor entries in both devices' files collapse to one.
- **Merged views are `merge=ours`** (never git-line-merged) + regenerated by the `post-merge` hook. The
  other device's entries come in via `/pickup absorb-all` reading `git show <ref>:<path>` — deterministic,
  independent of the working tree, and works even when the other device is still on the OLD single-file format.
- **Logs union in every merge mode** (`both`/`theirs`/`ours`) — the mode only changes which app *code* is adopted.
- **component-sync-log stays `merge=union`** — it's a markdown TABLE + prose; markers between rows would
  break the table, and union already keeps both devices' rows. Judgment call, documented.

## Gotchas fixed during build (all caught by tests)
- Windows cp1252 console can't encode `←`/`→` in status prints → force UTF-8 stdout + ASCII text.
- git-bash mangles `ref:sub/dir/file` blob specs (`:`+`/` → Windows path) → `MSYS_NO_PATHCONV=1`.
- HANDOFF timestamps contain a space → marker `ts` must be a whitespace-free sort key (`sortable()`).
- `migrate`/`absorb` must route each entry to ITS OWN device segment (self-tagged HANDOFF), not dump all
  into one segment.
