# Frontend Spec — Technical Architecture

## Hook Detection Logic

```
PreToolUse fires on Write|Edit
  │
  ├─ Read file_path from tool input
  ├─ Match against: apps/web/src/**/*.tsx, *.ts, *.css, packages/ui-tokens/
  │
  ├─ NOT a frontend file → exit 0 (allow, no action)
  │
  ├─ IS a frontend file → check for frontend_spec.md:
  │   ├─ Check .adr/orchestration/*/frontend_spec.md (any session)
  │   ├─ Check .adr/orchestration/frontend_spec.md (root)
  │   │
  │   ├─ FOUND → exit 0 (allow, spec exists)
  │   │
  │   └─ NOT FOUND → auto-generate:
  │       ├─ Detect design tokens (packages/ui-tokens/src/tokens.ts)
  │       ├─ Detect prototypes (.docs/planning/frontend/**/pass-*/*.html)
  │       ├─ Detect project context (CLAUDE.md, PRD, tech spec, task list)
  │       ├─ Generate frontend_spec.md at .adr/orchestration/
  │       ├─ Validate: are tokens + prototypes actually populated?
  │       │   ├─ YES → complete spec, no warnings
  │       │   └─ NO → mark "ACTION REQUIRED", warn in output
  │       └─ exit 0 (allow, spec now exists)
```

## Auto-Generation Algorithm

```python
def auto_generate_spec():
    # 1. Detect design tokens
    tokens = read("packages/ui-tokens/src/tokens.ts")
    primary = extract(tokens, "primary")    # e.g. '#000000'
    secondary = extract(tokens, "secondary") # e.g. '#FF3366'
    accent = extract(tokens, "accent")       # e.g. '#00FF88'

    # 2. Detect prototypes
    prototypes = glob(".docs/planning/frontend/**/pass-*/*.html")
    page_table = [(basename(p), p) for p in prototypes]

    # 3. Detect project context
    context = {
        "claude_md": exists("CLAUDE.md") or exists(".claude/CLAUDE.md"),
        "prd": find(".adr/orchestration/*/prd.md"),
        "tech_spec": find(".adr/orchestration/*/technical_requirements.md"),
        "task_list": find(".adr/orchestration/*/primary_task_list.md"),
        "adr_exists": exists(".adr/orchestration/"),
    }

    # 4. Determine completeness
    has_design_system = bool(primary) or bool(prototypes)

    # 5. Generate spec
    spec = render_template("frontend_spec_template.md", {
        colors, page_table, context, has_design_system
    })

    if not has_design_system:
        spec.mark_incomplete("ACTION REQUIRED")

    write(".adr/orchestration/frontend_spec.md", spec)
```

## Resolution Cascade

```
Agent needs frontend spec
  │
  ├─ 1. Session-level: .adr/orchestration/<SESSION>/frontend_spec.md
  │     └─ Most specific — overrides everything for this session
  │
  ├─ 2. Root-level: .adr/orchestration/frontend_spec.md
  │     └─ Default for sessions without their own spec
  │
  ├─ 3. Auto-generated by hook (if 1 and 2 missing)
  │     └─ Created at root level from detected design system
  │
  └─ 4. Incomplete spec with ACTION REQUIRED (if nothing detected)
        └─ Agent warned, should ask user for design input
```

## NL Parsing Rules (Skill)

When the skill processes a user message, it detects spec type from natural language:

```
URL detected (https://, http://, figma.com, etc.)
  → Type 2: External Reference
  → Extract URL, set match level from context ("exactly like" = 1:1, "inspired by" = loose)

Style keywords detected (round buttons, gradient, serif, dark theme, etc.)
  → Type 3: Chat-Prompted
  → Capture all style descriptions verbatim

Design pass referenced (pass-1, brutalism, liquid, etc.)
  → Type 4: Similarity Match
  → Extract pass name, parse divergence ("but with X" = diverge on X)

No design language + tokens exist
  → Type 1: Internal Prototype
  → Auto-detect from filesystem

No design language + no tokens
  → Ask batch questions (all at once):
    1. Reference site/image?
    2. Color preferences?
    3. Typography?
    4. Interaction style?
```

**Batch rule:** ALL questions in ONE prompt. Never ask one, wait, ask another.

**One-shot rule:** If user provides enough info, create spec immediately with zero questions.

## Mandatory Frontend Reading Order

Before writing ANY .tsx or .css file, agents must read in this order:

```
1. frontend_spec.md
   └─ Session-level, then root default

2. The referenced prototype/URL/spec from step 1
   └─ Read the actual HTML file, or screenshot the URL

3. CLAUDE.md
   └─ Project conventions, architecture, design system info

4. .adr/orchestration/<SESSION>/prd.md (if exists)
   └─ What's being built and why

5. .adr/orchestration/<SESSION>/technical_requirements.md (if exists)
   └─ Tech constraints (framework, tokens, responsive breakpoints)

6. The specific prototype HTML for the page being built
   └─ e.g. school-profile.html if building the school profile
```

## Project Context Detection

The hook and skill detect these project files:

```bash
# Scanned locations
CLAUDE.md                                    # or .claude/CLAUDE.md
.adr/orchestration/                          # ADR root
.adr/orchestration/*/prd.md                  # Any session PRD
.adr/orchestration/*/technical_requirements.md  # Any tech spec
.adr/orchestration/*/primary_task_list.md    # Any task list
packages/ui-tokens/src/tokens.ts             # Design tokens
.docs/planning/frontend/                     # Prototype directory
```

Results populate the `## Project Context` section:
```markdown
## Project Context
- **CLAUDE.md:** exists at .claude/CLAUDE.md
- **PRD:** .adr/orchestration/3_FRONTEND_DEVELOPMENT/prd.md
- **Tech Spec:** .adr/orchestration/3_FRONTEND_DEVELOPMENT/technical_requirements.md
- **Task List:** .adr/orchestration/9_DYNAMIC_NAV/primary_task_list.md
- **Design Tokens:** packages/ui-tokens/src/tokens.ts (primary=#000000, secondary=#FF3366)
- **ADR Status:** orchestration exists with 10 session subfolders
```

Or when nothing exists:
```markdown
## Project Context
- **CLAUDE.md:** not found
- **PRD:** not yet created
- **Tech Spec:** not yet created
- **Task List:** not yet created
- **Design Tokens:** not found
- **ADR Status:** ADR not set up yet — this spec was created standalone
```

## Self-Describing Header

Every `frontend_spec.md` starts with an HTML comment that any agent can read:

```html
<!--
  FRONTEND SPECIFICATION REFERENCE

  What this file is:
    Design reference for all frontend/UI work in this project.
    Any agent building UI MUST read this file before writing .tsx/.css.

  Project context (read these if they exist):
    - CLAUDE.md — project conventions and architecture
    - .docs/planning/ — PRD, feature list, user stories
    - .adr/orchestration/ — session task lists and phase plans

  How to use:
    1. Read Spec Source Type to know where the design reference lives
    2. Read the referenced prototype/URL/spec before building
    3. Follow the Coherence Rules during implementation
    4. After building, validate output matches this spec

  If this file says "ACTION REQUIRED":
    Auto-generated but no design system detected. Provide requirements.
-->
```
